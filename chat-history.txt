Build a Chrome extension that analyzes website copy and provides actionable
feedback to improve messaging, clarity, and engagement.

- Extract copy from the current webpage (headlines, CTAs, value propositions,

- Provide specific, actionable improvement suggestions

  ## Key Implementation Details

---

USER: Why the truncation

A: LLM context limit. Big pages have thousands of words — send all, hit token ceiling, waste cost, slow response.
8000 chars ≈ 1500-2000 tokens, enough for meaningful analysis without burning budget.

---

USER: Since we are creating nodes on langgraph for parallel execution can't we section it separately and analyze later aggregating?

A: Valid idea, but category nodes are concern-based not section-based. cta node only cares about ctas[], valueProp only cares about headlines/valueProps. bodyText mainly feeds readability, tone, jargon.
Chunking bodyText + parallel sub-nodes adds complexity: dynamic fan-out, merge logic per category, more LLM calls = more cost + latency.
8000 chars covers most landing pages fully. Truncation only hits very long pages where readability analysis on a sample is still representative.
Decision: keep truncation for now, note "large-page chunking" in NOTES.md production concerns (already planned).

- setCached(key, report): chrome.storage.local.set with timestamp
- sendProgress(stage): chrome.runtime.sendMessage PROGRESS broadcast (swallows error if popup closed)
- onInstalled: console.log stub retained
- onMessage listener:
  - Guards: ANALYZE_PAGE type check; VITE_GEMINI_API_KEY present; active tab exists; URL not restricted

  - chrome.scripting.executeScript({ func: extract }) → parse with ExtractedCopySchema.safeParse

  - sendProgress('analyzing')
  - buildGraph().invoke({ extracted }) → sendProgress('aggregating') → AnalysisReportSchema.safeParse
  - Error mapping: apiKey strings → MISSING_KEY; other throws → LLM_ERROR
  - setCached + sendProgress('done') on success
  - Returns true to keep async channel open for async sendResponse
